# Claude Code — 学んだこと（Lessons Learned）

> 本ファイルはClaude Codeがこのプロジェクトで学んだ教訓・パターン・ユーザーの好みを記録する。
> 新しい環境・セッションでも一貫した振る舞いを維持するために参照する。

---

## 1. ユーザーの作業スタイル

### 意思決定
- 選択肢を提示すると即断する。迷った時は「具体的に教えて」と聞いてくる
- 技術的な判断はClaudeに委ねることが多いが、業務要件は自分で決める
- 「分からない」を正直に言う。分からないまま進めることを嫌う

### コミュニケーション
- 簡潔な回答を好む。長い説明より表形式のまとめが好まれる
- 質問には番号付きで回答する（Q1: a、Q2: b のスタイル）
- 「おなしゃす」「よし」「了解」で承認を示す

### 作業の進め方
- 大量の作業は「なるべく並列で」を常に指示する
- レビュー→修正→レビューのサイクルを重視する
- 修正したらレビュー記録票も必ず更新することを求める
- HTML再生成を忘れずに行うことを期待する
- 一時ファイルやゴミファイルの後始末を気にする
- **PRのマージは必ずユーザー自身が行う。Claudeが勝手にマージしてはいけない**
- 長時間の自律作業を任せることがある（「お風呂入ってくるわ」等）

## 2. プロジェクト固有のルール

### ドキュメント管理
- **SSOTルール**が最重要。1つの事実は1箇所にだけ定義する
- 設計書間の矛盾を非常に嫌う。矛盾を見つけたら即座に修正を求められる
- レビュー記録票は必ず `reviews/` サブフォルダに配置する（スキルにも明記済み）
- ファイル命名は小文字統一

### ID管理
- `_id-registry.md` が全ID体系のSSoT。新規ID追加時は必ずここを先に更新する
- `build-docs.js --validate` でID整合性を自動検証する仕組みを構築済み
- `_standard-api.md` と `_standard-screen.md` にはID一覧を置かない（参照リンクのみ）

### HTMLビルド
- ドキュメント修正後は `node docs/scripts/build-docs.js` でHTML再生成する
- `--validate` オプションで事前検証してからビルドするのが理想
- spec.mdとbuild-docs.jsは常に同期させる

### コミット
- コミットメッセージは日英併記（`docs: 日本語 / English`）
- Co-Authored-By を含める

## 3. 技術的な教訓

### Mermaidフローチャート
- ノード内の `()` はノード形状と衝突する。`UNIQUE(a, b)` → `UNIQUE: a, b` に置換
- `{}` ダイヤモンドノード内の `{id}` はURL変数でも構文エラーになる。`:id` を使う
- `\n` は多くの場合動作するが、特殊文字と組み合わせると壊れる。`<br/>` が安全
- 正規表現（`[A-Z]`, `\\d`, `$`）はMermaidノード内で使えない。説明文に簡略化する
- `mermaid-cli` (`npx mmdc`) で事前検証可能

### 設計書間の整合性
- ステータス値の不一致が最も頻発する問題。Enum定義（08-common-infrastructure.md）、data-model、API設計書、画面設計書の4箇所を常に同期させる必要がある
- 環境変数名の不一致も頻発（`JWT_SECRET` vs `JWT_SECRET_KEY`、`CORS_ALLOWED_ORIGINS` vs `FRONTEND_URL`）
- APIパスの `/master/` プレフィックスの有無が画面設計書とAPI設計書でずれやすい
- エラーコード体系は英語定数名（`DUPLICATE_CODE`等）に統一済み

### OpenAPI定義
- OpenAPI 3.1のnullable構文は `type: ['string', 'null']`（`nullable: true` は3.0の書き方）
- Redocly CLIでバリデーション: `npx @redocly/cli lint openapi/wms-api.yaml`
- 共通コンポーネント（ResourceId, PageParam, SizeParam, ReportFormat, ValidationError, Unauthorized, Forbidden, OptimisticLockConflict）を積極的に再利用する
- レポートAPIのように同一パターンのエンドポイントが多い場合、レビューで403等の共通レスポンスの付け忘れが頻発する

### 並列作業の注意点
- 同一ファイルを複数エージェントが同時修正するとコンフリクトする。ファイル単位でエージェントを分けること
- 大きなファイル（API-10-report.md、06-infrastructure-architecture.md等）は1エージェントに集中させる
- 設計書レビューは13 API分を並列エージェントで実行可能（各エージェントが設計書1つ+OpenAPIの該当部分を読む）

## 4. 避けるべきパターン

| パターン | 理由 | 代替策 |
|---------|------|--------|
| ID一覧を複数箇所に書く | 必ず乖離する | `_id-registry.md` をSSOTとし、他は参照リンク |
| テーブル定義をAPI設計書に複製する | data-modelと乖離する | 「data-modelを参照」リンクのみ |
| カバレッジ目標を複数箇所に書く | 値が不一致になる | `00-test-plan.md` をSSOTとする |
| レビュー記録票をreviews/外に置く | ビルドスクリプトが検出できない | 必ず `reviews/` サブフォルダに配置 |
| 一時ファイルを放置する | ユーザーに指摘される | 作業完了後に即削除 |
| HTMLビルドを忘れる | ユーザーが最新のドキュメントを確認できない | 修正後は必ずビルド |

## 5. 確定した設計判断の要約

| 判断 | 結論 | 理由 |
|------|------|------|
| API方式 | API First（OpenAPI → コード生成） | フロント・バックエンド並行開発、型一致保証 |
| 出荷ステータス | `ALLOCATED`（PICKING_INSTRUCTEDは廃止） | 引当完了=ピッキング指示済み |
| 認証方式 | Managed Identity（Blob Storage, ACR） | シークレット管理不要 |
| パスワードリセットトークン | DELETE方式（usedフラグなし） | セキュリティ、シンプルさ |
| Composableパターン | A案（画面単位で完結、共通抽出しない） | AI向き、使い捨てが効く |
| エラーハンドリング | おまかせパターン（インターセプターで共通処理） | 画面側コード削減 |
| 楽観ロック競合時UX | シンプル通知（再読み込みボタン） | WMSの同時編集頻度は低い |
| タイムゾーン | JST固定 | 国内向けShowCase |
| テストカバレッジ | C0/C1: 100%、C2: 対象外 + 品質ルール3件 | C2は組み合わせ爆発。品質ルールでAIの手抜き防止 |
| SameSite | Lax | CSRF対策とユーザビリティのバランス |
| バッチSecurityContext | 実行者ユーザーID | 監査証跡 |
| 営業日カレンダー | 管理しない（休日は手動3回実行） | ShowCase規模では不要 |

## 6. 開発フェーズの学び

### 開発ワークフロー
- Issue作成時にタスク一覧を詳細化して記載する（受入条件ではなくタスク一覧）
- PR作成後は対象領域の専門家レビューを実施し、結果をPRコメントに追記する
- レビュー指摘は1件ごとに変更→コミット→Push→PRコメント追記のサイクルで対応する
- 自身の変更以外のドキュメント修正や他領域のコード変更は別Issueを作成する
- タスク完了ごとにIssueのタスク一覧にチェックをつけて進捗を可視化する

### OpenAPIと設計書の整合性
- 設計フェーズ完了後、OpenAPI定義を作成したら必ず全設計書との整合性レビューを行う
- 頻出する差分パターン:
  - エンドポイントパスの命名不一致（shipping-list vs delivery-list）
  - 403 Forbiddenレスポンスの付け忘れ（特に大量エンドポイント作成時）
  - レスポンススキーマのrequired配列の未定義
  - バリデーション制約（minLength等）の未定義
  - 設計書内部の矛盾（同一ドキュメント内で異なる値を記載）
