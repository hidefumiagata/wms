# ドキュメントレビュータスク

## 対象ファイル
`$ARGUMENTS`

## あなたの役割
対象ファイルの内容に最も適した専門家として振る舞ってください。
- API設計書 → Java バックエンド REST API 設計スペシャリスト
- 画面設計書 → フロントエンド UI/UX 設計スペシャリスト
- アーキテクチャ設計書 → クラウドアーキテクト
- 機能要件定義書 → ビジネスアナリスト
- データモデル定義書 → データベース設計スペシャリスト

## レビュー手順

### Step 1: 対象ファイルの読み込み
対象ファイル（`$ARGUMENTS`）を読んで内容を把握する。

### Step 2: 関連ドキュメントの特定と読み込み
対象ファイルの種類に応じて、以下を参照して照合する：
- `docs/functional-requirements/` — 機能要件定義書（全7ファイル）
- `docs/data-model/` — データモデル定義（全4ファイル）
- `docs/functional-design/08-api-overview.md` — API概要・エラーコード一覧（API設計書の場合）
- `docs/architecture-blueprint/` — アーキテクチャブループリント
- `docs/functional-design/` 内の関連ファイル

### Step 3: レビュー実施
以下の観点でチェックする：
- **要件との整合性**: 機能要件定義書の記述と矛盾していないか
- **他設計書との整合性**: データモデル・API概要・画面設計との矛盾
- **内部一貫性**: ファイル内でのフィールド名・ステータス値・型の統一
- **抜け漏れ**: 要件に定義されているが設計書に反映されていない項目
- **誤記**: タイプミス、コピペミスなど

### Step 4: レビュー記録票の作成・更新
対象ファイルのパスからレビュー記録票のファイル名を決定する。**必ず `reviews/` サブフォルダに配置する**:
- `docs/{カテゴリ}/` 配下のファイル → `docs/{カテゴリ}/reviews/review-records-[ファイル名].md`
  - 例: `docs/functional-design/API-06-inbound.md` → `docs/functional-design/reviews/review-records-api-06-inbound.md`
  - 例: `docs/test-specifications/TST-INB-inbound.md` → `docs/test-specifications/reviews/review-records-tst-inb-inbound.md`
  - 例: `docs/architecture-design/04-backend-architecture.md` → `docs/architecture-design/reviews/review-records-04-backend-architecture.md`
- **ファイル名は小文字に統一する**
- `reviews/` フォルダが存在しない場合は作成する

レビュー記録票に以下の形式で記録する：

```markdown
# レビュー記録票: [対象ファイル名]

**レビュー実施日**: YYYY-MM-DD
**対象ファイル**: [パス]
**レビュアー**: Claude Code (自動レビュー)

## サマリー

| 分類 | 件数 |
|------|:----:|
| 対象ファイル修正済み | N |
| 要対応（他ドキュメント） | N |
| 指摘なし | N |
| **合計** | **N** |

## 詳細指摘テーブル

| No | 対象箇所 | 参照ドキュメント | 指摘内容 | 分類 | 対応内容 |
|----|---------|-----------------|---------|------|---------|
| 1 | ... | ... | ... | 対象ファイル修正済み / 要対応 / 指摘なし | ... |
```

### Step 5: 対象ファイルの修正
- **対象ファイル（`$ARGUMENTS`）に問題がある場合**: 即時修正する
- **他のドキュメントに問題がある場合**: 修正せず、レビュー記録票に「要対応」として記載するのみ

修正を行った場合は、レビュー記録票のサマリーを「対象ファイル修正済み」として更新する。

### Step 6: 完了報告
以下を報告する：
- レビューした観点の数
- 発見した問題の数と内訳（修正済み / 要対応）
- 特に重要な指摘事項のサマリー
