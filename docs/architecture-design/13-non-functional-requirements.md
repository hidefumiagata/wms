## 1. パフォーマンス要件設計

### 1.1 応答時間目標

| 分類 | 目標値 | 測定ポイント | 対象操作例 |
|------|--------|-------------|-----------|
| **API応答（通常CRUD操作）** | 500ms以内 | バックエンドAPI処理時間（Controller受信〜レスポンス送信） | マスタ登録・更新、在庫照会、一覧検索 |
| **API応答（一覧検索・ページネーション付き）** | 1秒以内 | 同上 | 入荷予定一覧、受注一覧、在庫一覧（ページング付き） |
| **API応答（集計・レポート系）** | 5秒以内 | 同上 | 日次集計レポート生成、PDF出力 |
| **画面初期表示（SPA）** | 3秒以内 | ブラウザでの初回ロード完了（FCP） | ログイン画面、ダッシュボード |
| **画面遷移（SPA内ルーティング）** | 300ms以内 | Vue Routerによるコンポーネント切り替え | メニュー間遷移 |
| **バッチ処理（日替処理全体）** | 30分以内 | 5ステップの合計実行時間 | BAT-001 日替処理 |

### 1.2 スループット目標

| 項目 | 目標値 | 前提条件 |
|------|--------|---------|
| **同時APIリクエスト処理数** | 50リクエスト/秒 | prd環境・ピーク時 |
| **CSV取り込み処理** | 10,000行/分 | IFX-001/IFX-002 外部連携I/F |
| **棚卸実数入力** | 2,000明細/棚卸 | 1回の棚卸操作の上限明細数 |
| **PDF生成** | 5秒以内/帳票 | 1帳票あたりの生成時間 |

### 1.3 パフォーマンス実現手段

#### 1.3.1 バックエンドAPI最適化

| 対策 | 実現方法 | 対象 |
|------|---------|------|
| **ページネーション** | Spring Data JPA `Pageable` による遅延ロード。デフォルトページサイズ20件、最大100件 | 全一覧系API |
| **N+1クエリ防止** | `@EntityGraph` または `JOIN FETCH` による関連エンティティの一括取得 | 入荷明細・受注明細の一覧取得 |
| **インデックス設計** | 検索条件に使用するカラムへのBTreeインデックス付与 | ステータス、営業日、倉庫コード、商品コード |
| **コネクションプール** | HikariCP。最大接続数: dev=5, prd=10 | PostgreSQL接続 |
| **非同期処理** | PDF生成は `@Async` で非同期実行。生成完了後にダウンロードURLを返却 | レポートPDF出力 |

#### 1.3.2 フロントエンド最適化

| 対策 | 実現方法 | 効果 |
|------|---------|------|
| **コード分割** | Vite の動的インポート（`() => import(...)`）によるルート単位のコード分割 | 初期ロードサイズ削減 |
| **アセット最適化** | Vite ビルドによる Tree Shaking、CSS/JS の minify、gzip 圧縮 | 転送サイズ削減 |
| **APIレスポンスキャッシュ** | マスタデータ（商品・取引先・倉庫）のフロントエンドキャッシュ（Pinia Store、TTL: 5分） | API呼び出し削減 |
| **仮想スクロール** | Element Plus `el-table-v2`（仮想化テーブル）の利用。1,000行以上の一覧表示時に適用 | 大量データ表示のDOM負荷軽減 |
| **デバウンス** | 検索入力に300msのデバウンス処理を適用 | 過剰なAPI呼び出し抑制 |

#### 1.3.3 データベース最適化

| 対策 | 実現方法 | 対象 |
|------|---------|------|
| **インデックス戦略** | 主要検索条件への複合インデックス。`EXPLAIN ANALYZE` で実行計画を検証 | 後述の「9. 容量設計」参照 |
| **パーティショニング** | 本フェーズでは不採用。データ量増加時の将来施策として検討 | トランザクションテーブル |
| **トランデータアーカイブ** | 日替処理（BAT-001）で完了済み2か月超のデータをバックアップテーブルに移行し、主テーブルの肥大化を抑制 | 入荷・出荷・在庫変動テーブル |
| **クエリタイムアウト** | `spring.jpa.properties.jakarta.persistence.query.timeout=30000`（30秒） | 全クエリ |

### 1.4 パフォーマンス監視

| 項目 | 監視方法 | 閾値 |
|------|---------|------|
| **API応答時間** | Application Insights による自動計測 | 平均3秒超過/5分でアラート |
| **スロークエリ** | PostgreSQL `log_min_duration_statement=1000`（1秒以上のクエリをログ出力） | 1秒以上のクエリ |
| **メモリ使用量** | Azure Monitor Container Apps メトリクス | 80%超過でアラート |
| **CPU使用率** | Azure Monitor Container Apps メトリクス | 80%超過でアラート |

## 2. 可用性設計

> 可用性の基本方針は [06-infrastructure-architecture.md](../architecture-blueprint/06-infrastructure-architecture.md) のインフラ構成に従う。

### 2.1 SLA目標

| 環境 | 目標稼働率 | 条件 |
|------|-----------|------|
| **prd（稼働中）** | 99.5%以上 | 計画停止（Terraform Destroy/Deploy）を除く |
| **dev** | SLA対象外 | 開発・テスト用途。Terraform Deploy/Destroy で必要時のみ起動 |

### 2.2 業務稼働時間帯

| 項目 | 内容 |
|------|------|
| **業務稼働時間帯** | 平日 7:00〜22:00（JST） |
| **計画停止枠** | 月次メンテナンス最大2時間（業務時間外に実施） |
| **計画停止の事前通知** | 5営業日前までにメール通知 |

### 2.3 障害復旧目標

| 項目 | 目標値 | 実現方法 |
|------|--------|---------|
| **RTO（フロント/API）** | 数分 | Azure Front Door による自動フェイルオーバー（prd のみ） |
| **RTO（DB復旧）** | 数時間 | 手動対応（ポイントインタイムリストア） |
| **RPO** | 最大1時間 | Geo-redundant backup 基点（prd のみ） |

### 2.4 冗長構成（prd環境）

| コンポーネント | Active（Japan East） | Standby（Japan West） | フェイルオーバー方式 |
|--------------|---------------------|----------------------|-------------------|
| **フロントエンド** | Blob Storage（Static Website） | GRS 自動レプリカ | Front Door 自動切替 |
| **バックエンドAPI** | Container Apps min:1 max:5 | Container Apps min:0 max:5 | Front Door ヘルスプローブ失敗時に自動切替 |
| **DB** | PostgreSQL Flexible Server（B1ms） | なし（East のみ） | 手動リストア（Geo-redundant backup） |
| **ストレージ** | Blob Storage（GRS） | Japan West へ自動レプリケーション | GRS 自動フェイルオーバー |
| **コンテナレジストリ** | ACR Basic | Geo-replication（East + West） | 自動レプリケーション |

### 2.5 ヘルスチェック設計

| エンドポイント | チェック内容 | 間隔 | 用途 |
|--------------|------------|------|------|
| `GET /actuator/health` | アプリケーション起動状態 + DB接続確認 | 30秒 | Container Apps リブネスプローブ |
| `GET /actuator/health/readiness` | アプリケーション準備完了状態 | 10秒 | Container Apps レディネスプローブ |
| `GET /actuator/info` | バージョン・ビルド情報 | - | 運用確認用（プローブ対象外） |

Front Door ヘルスプローブは `/actuator/health` を使用し、3回連続失敗でバックエンドを「不健全」と判定する。

### 2.6 グレースフルシャットダウン

| 項目 | 設定 |
|------|------|
| **シャットダウン猶予時間** | 30秒（`spring.lifecycle.timeout-per-shutdown-phase=30s`） |
| **処理中リクエストの扱い** | シャットダウンシグナル受信後、新規リクエストを拒否し、処理中のリクエストが完了するまで最大30秒待機 |
| **バッチ処理中のシャットダウン** | バッチ処理はステップ単位で進捗を記録しているため、再起動後に未完了ステップから再開可能 |

## 3. スケーラビリティ設計

### 3.1 水平スケーリング（バックエンド）

| 環境 | min replicas | max replicas | スケールトリガー | スケールアウト条件 |
|------|-------------|-------------|---------------|----------------|
| **dev** | 0 | 3 | HTTPリクエスト数 | 同時リクエスト10以上/レプリカ |
| **prd East** | 1 | 5 | HTTPリクエスト数 | 同時リクエスト20以上/レプリカ ※暫定値。負荷テストフェーズで実測し、適切な閾値に調整する |
| **prd West** | 0 | 5 | HTTPリクエスト数 | Front Door からのルーティング発生時 |

#### スケーリング時の考慮事項

| 項目 | 対策 |
|------|------|
| **セッション管理** | JWT（httpOnly Cookie）のためサーバーサイドセッション不要。ステートレス設計によりレプリカ間のセッション共有が不要 |
| **DBコネクション管理** | HikariCP のコネクションプール上限をレプリカ数を考慮して設定（prd: max 10/レプリカ × max 5レプリカ = 50接続。PostgreSQL B1ms の max_connections=50 に収まる設計） |
| **ファイルアップロード** | Blob Storage への直接書き込みのため、レプリカ間でのファイル共有問題なし |
| **バッチ処理の排他** | 日替処理はDB上の処理済みフラグで二重実行を防止。複数レプリカから同時実行されても冪等 |

### 3.2 垂直スケーリング

| コンポーネント | 現行スペック | スケールアップ基準 | スケールアップ先 |
|--------------|------------|----------------|----------------|
| **Container Apps** | vCPU: 0.5, Memory: 1.0Gi | CPU/メモリ常時80%超過 | vCPU: 1.0, Memory: 2.0Gi |
| **PostgreSQL** | B1ms（1 vCore, 2GiB RAM） | 接続数上限、CPU常時80%超過 | B2s（2 vCore, 4GiB RAM） |

### 3.3 データ増加への対応

| 対策 | 内容 | 実行タイミング |
|------|------|--------------|
| **トランデータアーカイブ** | 完了済み2か月超のデータをバックアップテーブルに移行 | 日替処理（BAT-001）で自動実行 |
| **インデックスメンテナンス** | `REINDEX` によるインデックス再構築 | 月次メンテナンス時に手動実行 |
| **VACUUM ANALYZE** | テーブル統計情報の更新と不要領域の回収 | PostgreSQL autovacuum に委任（デフォルト設定） |

## 4. 信頼性設計

### 4.1 データ整合性

| 対策 | 実現方法 | 適用範囲 |
|------|---------|---------|
| **トランザクション管理** | Spring `@Transactional` によるACID保証。Service層メソッド単位でトランザクション境界を設定 | 全データ更新操作 |
| **楽観的ロック** | JPA `@Version` カラムによる更新競合検出。競合時は `OptimisticLockConflictException` をスロー | マスタデータ更新、入荷/出荷ステータス更新 |
| **一意制約** | DB の UNIQUE 制約 + アプリケーション層での事前チェック（`DuplicateResourceException`） | 各種コード（商品コード、倉庫コード等） |
| **外部キー制約** | PostgreSQL の FOREIGN KEY 制約で参照整合性を保証 | テーブル間の参照関係 |
| **NOT NULL制約** | 必須カラムへのNOT NULL制約 | 全テーブルの必須項目 |

### 4.2 論理削除方針

| 項目 | 内容 |
|------|------|
| **対象** | 全マスタデータ（商品・取引先・倉庫・棟・エリア・ロケーション・ユーザー） |
| **実装** | `is_active` フラグ（boolean）。`false` で論理削除状態 |
| **参照制約** | 無効化したマスタは新規トランザクションの選択候補に表示しない。既存トランザクションの参照は維持 |
| **物理削除** | 一切行わない。データ整合性とトレーサビリティを優先 |

### 4.3 障害復旧設計

| 障害パターン | 検知方法 | 復旧手順 | 想定復旧時間 |
|------------|---------|---------|------------|
| **アプリケーション障害** | ヘルスプローブ失敗 | Container Apps 自動再起動 | 1〜2分 |
| **リージョン障害（prd）** | Front Door ヘルスプローブ | 自動フェイルオーバー（Japan West） | 数分 |
| **DB障害** | 接続エラーログ検知 | ポイントインタイムリストア（手動） | 数時間 |
| **データ不整合** | 業務運用中の発見 | トランザクションログ調査 → 手動データ修正 | ケースバイケース |
| **バッチ処理失敗** | バッチ実行結果エラー | ステップ単位で再実行（冪等設計） | 数分〜30分 |

### 4.4 バックアップ設計

| 対象 | 方式 | 保持期間 | リストア方法 |
|------|------|---------|------------|
| **PostgreSQL** | Azure Flexible Server 自動バックアップ | 7日間 | ポイントインタイムリストア |
| **PostgreSQL（prd）** | Geo-redundant backup | 7日間 | クロスリージョンリストア |
| **Blob Storage（iffiles）** | LRS（dev）/ GRS（prd） | 無期限 | Azure Portal から復元 |
| **業務データアーカイブ** | バックアップテーブルへの複製（日替処理） | 無期限 | SQLクエリで参照・復元 |
| **Terraform State** | Azure Blob Storage（専用サブスクリプション） | 無期限 | Blob バージョニングで復元 |

### 4.5 二重実行防止

| 処理 | 防止方法 |
|------|---------|
| **日替処理（BAT-001）** | 対象営業日の処理済みフラグをDBで管理。全ステップ完了済みの場合はエラー返却 |
| **入庫確定** | 入荷明細のステータス遷移チェック + 楽観的ロック |
| **出荷確定** | 受注明細のステータス遷移チェック + 楽観的ロック |
| **在庫引当** | 有効在庫数（`quantity - allocated_qty`）の再計算 + 楽観的ロック |

## 5. 保守性設計

### 5.1 コード品質基準

| 項目 | 基準 | 検証方法 |
|------|------|---------|
| **コーディング規約** | Google Java Style Guide（バックエンド）、ESLint + Prettier（フロントエンド） | CI/CDパイプラインでの自動チェック |
| **メソッド行数** | 1メソッドあたり50行以内を推奨 | コードレビュー |
| **クラスの責務** | 単一責任の原則（SRP）に従う。Controller / Service / Repository の責務を明確に分離 | コードレビュー |
| **コメント** | Javadoc: public API に必須。インラインコメント: 「なぜ」を記述（「何」はコードで表現） | コードレビュー |
| **テストカバレッジ** | 後述「8. テスト要件設計」参照 | JaCoCo / Vitest coverage |

### 5.2 アーキテクチャ規約

| 規約 | 内容 |
|------|------|
| **レイヤー構成** | Controller → Service → Repository の3層構成を厳守。レイヤーをスキップするアクセスは禁止 |
| **モジュール間依存** | 異なるモジュール間は Service 層を介した間接参照のみ許可。Repository の直接参照は禁止 |
| **DTO設計** | Controller 層は DTO（Request/Response）で受け渡し。Entity をレスポンスとして直接公開しない |
| **例外処理** | Service層でカスタム例外をスロー → GlobalExceptionHandler で一元変換。詳細は [08-common-infrastructure.md](../architecture-blueprint/08-common-infrastructure.md) 参照 |
| **ログ出力** | 構造化JSONログ（traceId付き）。PIIマスキング必須。詳細は [08-common-infrastructure.md](../architecture-blueprint/08-common-infrastructure.md) 参照 |

### 5.3 ドキュメント基準

| ドキュメント | 基準 |
|------------|------|
| **APIドキュメント** | OpenAPI 3.0（Swagger）で自動生成。Controller のアノテーションで定義 |
| **データベーススキーマ** | Flyway マイグレーションスクリプトで管理。手動のDDL変更は禁止 |
| **設計ドキュメント** | SSOTルールに従い、1つの事実は1箇所にだけ定義。詳細は [CLAUDE.md](../../CLAUDE.md) 参照 |
| **コミットメッセージ** | 日英併記。変更内容を簡潔に記述 |

### 5.4 デプロイ設計

| 項目 | 内容 |
|------|------|
| **CI/CDパイプライン** | GitHub Actions |
| **ビルド** | Gradle（バックエンド）、Vite（フロントエンド） |
| **コンテナイメージ** | ACR にプッシュ。タグはGitコミットハッシュ |
| **デプロイ方式** | Container Apps のリビジョン更新（ローリングアップデート） |
| **ロールバック** | 前リビジョンへの切り替え（Container Apps の機能） |
| **環境変数管理** | Container Apps 環境変数 + Terraform variables |
| **DBマイグレーション** | Flyway。アプリケーション起動時に自動実行 |

## 7. 互換性設計

### 7.1 ブラウザ対応

| 項目 | 対応範囲 |
|------|---------|
| **対応ブラウザ** | Google Chrome 最新版、Microsoft Edge 最新版 |
| **非対応** | Firefox、Safari、Internet Explorer、モバイルブラウザ |
| **対応解像度** | 1280×768 以上 |
| **対応デバイス** | デスクトップブラウザのみ（タブレット・スマートフォンはスコープ外） |

### 7.2 API後方互換性

| 項目 | 方針 |
|------|------|
| **APIバージョニング** | URLパスに `/api/v1/` を含むバージョニング方式 |
| **破壊的変更の扱い** | 新バージョン（`/api/v2/`）を追加し、旧バージョンは一定期間（3リリースまたは6か月）並行稼働 |
| **非破壊的変更** | レスポンスへのフィールド追加、オプショナルなリクエストパラメータ追加は後方互換と見なす |
| **廃止通知** | レスポンスヘッダー `Deprecation` で廃止予定を通知 |

### 7.3 データ互換性

| 項目 | 方針 |
|------|------|
| **DBスキーマ変更** | Flyway マイグレーションで管理。破壊的変更（カラム削除等）は段階的に実施 |
| **CSV I/Fフォーマット** | ヘッダー行の有無・カラム順序はバージョン管理。フォーマット変更時は旧フォーマットの並行受付期間を設ける |
| **タイムゾーン** | JST（Asia/Tokyo）固定。DB格納時は `TIMESTAMP WITH TIME ZONE` を使用 |
| **文字コード** | UTF-8 統一。CSV I/Fは BOM付きUTF-8 も受け入れ可能 |

## 8. テスト要件設計

### 8.1 テスト種別とカバレッジ目標

| テスト種別 | テストツール | カバレッジ目標 | 対象 |
|-----------|------------|-------------|------|
| **単体テスト（バックエンド）** | JUnit 5 + Mockito | 行カバレッジ 80%以上 | Service層の業務ロジック |
| **単体テスト（フロントエンド）** | Vitest + Vue Test Utils | 行カバレッジ 70%以上 | コンポーネントロジック、Piniaストア |
| **統合テスト** | Spring Boot Test + Testcontainers | 主要業務フロー100%カバー | API エンドポイント（Controller〜DB） |
| **E2Eテスト** | Playwright | 主要業務シナリオ100%カバー | 画面操作〜API〜DB の一貫テスト |

### 8.2 テスト範囲の優先度

| 優先度 | テスト対象 | 理由 |
|--------|----------|------|
| **最高** | 在庫数量計算（入庫・出庫・引当・ばらし・訂正） | データ不整合が業務に直結 |
| **最高** | ステータス遷移（入荷・出荷・棚卸・引当） | 不正遷移が業務フロー破綻に直結 |
| **高** | 認証・認可（ログイン・ロック・ロール制御） | セキュリティ要件 |
| **高** | 日替処理（バッチ：冪等性・二重実行防止） | 業務データの正確性 |
| **中** | マスタCRUD + バリデーション | 基本機能の品質 |
| **中** | CSV取り込み（バリデーション・エラーハンドリング） | 外部連携の信頼性 |
| **低** | PDF帳票出力 | 出力内容の正確性確認 |

### 8.3 テスト環境

| 項目 | 内容 |
|------|------|
| **CI環境** | GitHub Actions |
| **バックエンドテストDB** | Testcontainers（PostgreSQLコンテナ） |
| **フロントエンドテスト** | Node.js 環境（jsdom） |
| **E2Eテスト環境** | dev環境（Terraform Deploy） |
| **テストデータ** | Flyway テスト用マイグレーション + テストフィクスチャ |

### 8.4 CI/CDパイプラインでのテスト実行

```
PR作成 / コミットプッシュ
    ├── バックエンド
    │   ├── Gradle build（コンパイルチェック）
    │   ├── JUnit 5 単体テスト実行
    │   ├── 統合テスト実行（Testcontainers）
    │   └── JaCoCo カバレッジレポート生成
    ├── フロントエンド
    │   ├── ESLint + Prettier チェック
    │   ├── TypeScript 型チェック（tsc --noEmit）
    │   ├── Vitest 単体テスト実行
    │   └── カバレッジレポート生成
    └── カバレッジ閾値チェック（未達の場合はビルド失敗）
```

## 10. 運用要件設計

> 運用・監視の詳細設計は [11-monitoring-operations.md](../architecture-blueprint/11-monitoring-operations.md) を参照。本セクションでは運用要件としての方針と基準を記載する。

### 10.1 監視体制

| 監視分類 | 監視対象 | ツール | 参照先 |
|---------|---------|-------|--------|
| **アプリケーション監視** | ERRORログ、API応答時間、認証失敗 | Log Analytics + Azure Monitor アラート | [11-monitoring-operations.md](../architecture-blueprint/11-monitoring-operations.md) |
| **インフラ監視** | CPU、メモリ、ディスク、ネットワーク | Azure Monitor メトリクス | [11-monitoring-operations.md](../architecture-blueprint/11-monitoring-operations.md) |
| **DB監視** | 接続数、スロークエリ、ストレージ使用量 | Azure Monitor for PostgreSQL | [11-monitoring-operations.md](../architecture-blueprint/11-monitoring-operations.md) |

### 10.2 アラート設定

| アラート名 | 条件 | 重大度 | 通知方法 |
|-----------|------|--------|---------|
| **ERRORログ検知** | ERRORレベルのログが1件以上/5分 | 高 | メール |
| **APIレスポンス遅延** | 平均レスポンスタイム > 3秒/5分 | 中 | メール |
| **DB接続失敗** | DB接続エラーログ検知 | 高 | メール |
| **ログイン失敗多発** | ログイン失敗が10件以上/5分 | 中 | メール |
| **CPU使用率** | 80%超過/5分 | 中 | メール |
| **メモリ使用率** | 80%超過/5分 | 中 | メール |
| **ディスク使用率** | PostgreSQL ストレージ 80%超過 | 高 | メール |

### 10.3 ログ管理

| 項目 | 設定 |
|------|------|
| **ログ形式** | JSON構造化ログ（timestamp, level, logger, traceId, userId, message, module） |
| **ログレベル（prd）** | INFO（一時的にDEBUGに変更可能） |
| **ログレベル（dev）** | DEBUG |
| **ログ保存期間** | Log Analytics Workspace: 30日 |
| **PIIマスキング** | メールアドレス、電話番号を自動マスク |

### 10.4 定期運用タスク

| タスク | 頻度 | 実行方法 | 目的 |
|--------|------|---------|------|
| **日替処理** | 毎営業日 | バッチ管理画面から手動実行 | 営業日更新・集計・アーカイブ |
| **PostgreSQL停止/起動（dev）** | 必要時 | Azure CLI手動操作 | コスト削減 |
| **Terraform Destroy/Deploy（dev）** | 必要時 | GitHub Actions または手動 | コスト削減 |
| **インデックス再構築** | 月次 | SQL手動実行（`REINDEX`） | クエリパフォーマンス維持 |
| **ACR古いイメージ削除** | 月次 | Azure CLI（`az acr run --cmd purge`） | ストレージ節約 |
| **Dependabotアラート対応** | 随時 | GitHub PR レビュー + マージ | セキュリティパッチ適用 |

### 10.5 インシデント対応

インシデント対応フローは [11-monitoring-operations.md](../architecture-blueprint/11-monitoring-operations.md) を参照。

| 重大度 | 定義 | 対応時間目標 |
|--------|------|------------|
| **Critical** | サービス全体停止、データ損失リスク | 即時対応（1時間以内に初動） |
| **High** | 主要機能の障害、パフォーマンス著しく低下 | 4時間以内に初動 |
| **Medium** | 一部機能の障害、パフォーマンス低下 | 翌営業日対応 |
| **Low** | 軽微な不具合、改善要望 | 次回リリースで対応検討 |
