# 品質管理計画 (Quality Management Plan)

## 1. コーディング規約・コード品質

| 項目 | 方針 |
|------|------|
| **コーディング規約** | 技術スタック確定後に選定（Airbnb / Google等の標準規約を参照） |
| **Linter** | 採用言語に対応したLinterを使用、規約に沿った設定を行う |
| **Formatter** | 採用言語に対応したFormatterを使用 |
| **適用タイミング** | コミット時（pre-commitフック） |

## 2. コードレビュー

| 項目 | 方針 |
|------|------|
| **プロセス** | Claude CodeがレビューしたコードをGitHub Pull Requestでレビュー依頼 |
| **PR内容** | テスト結果 + レビュー指摘内容をPRコメントに記載 |
| **承認者** | プロジェクトオーナー（最終承認） |
| **マージ条件** | CI全通過 + オーナー承認 |

## 3. テスト方針

### 3.1 単体テスト

| 項目 | 内容 |
|------|------|
| **対象** | フロントエンド・バックエンド各コンポーネント |
| **カバレッジ基準** | C0 / C1 / C2 すべて100% |
| **外部接続** | すべてモック / スタブで代替 |
| **実行環境** | ローカル / CI自動実行 |

### 3.2 結合テスト

| 項目 | 内容 |
|------|------|
| **対象** | 画面 〜 API 〜 DB の結合 |
| **テストシナリオ** | 機能設計書のユースケースごとに作成 |
| **テストデータ** | ユースケース整理後に整備・事前投入 |
| **実行環境** | ローカル |

### 3.3 E2Eテスト

| 項目 | 内容 |
|------|------|
| **対象** | 本番相当データ量での結合テスト相当シナリオ |
| **テストシナリオ** | 結合テストと同等のユースケース |
| **実行環境** | 本番環境（デプロイ後） |

### 3.4 パフォーマンステスト

スコープ外（コスト上の理由）

## 4. CI/CD パイプライン

### 4.1 CI（継続的インテグレーション）

**実装タイミング：一通りの機能構築完了後（ShowCase用途として実装）**

| チェック項目 | 内容 |
|------------|------|
| Linterチェック | 規約違反の検出 |
| Formatterチェック | フォーマット違反の検出 |
| 単体テスト全通過 | 全テストケースのPass確認 |
| カバレッジチェック | C0 / C1 / C2 100%確認 |
| ビルドチェック | ビルド成功確認 |

### 4.2 CD（継続的デプロイメント）

| 環境 | トリガー | デプロイ方式 |
|------|---------|------------|
| **dev環境** | mainブランチへのマージ | 自動デプロイ |
| **prd環境** | 手動実行 | 手動デプロイ |

### 4.3 実行環境

GitHub Actionsを使用。開発フェーズ中はコスト抑制のためローカル実行を基本とし、機能完成後にGitHub Actionsとして整備する。

## 5. バージョン管理

**セマンティックバージョニング（SemVer）** を採用する。

```
MAJOR.MINOR.PATCH
例: 1.0.0 → 1.1.0（機能追加）→ 1.1.1（バグ修正）
```

| バージョン | 更新条件 |
|-----------|---------|
| MAJOR | 後方互換性のない変更 |
| MINOR | 後方互換性のある機能追加 |
| PATCH | バグ修正 |
