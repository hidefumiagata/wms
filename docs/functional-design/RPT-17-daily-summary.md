# レポート設計書 — 日次集計レポート

## 1. レポート概要

| 項目 | 内容 |
|------|------|
| **レポートID** | `RPT-17` |
| **レポート名** | 日次集計レポート |
| **用途** | 倉庫管理者が、日替処理完了後に1日の業務実績サマリーを確認するために使用する。入荷・出荷・在庫の概況と未処理アラートを一覧し、翌日の業務計画や経営報告の材料とする |
| **対応API** | `API-RPT-017`（[API-10-report.md](API-10-report.md) を参照） |
| **用紙サイズ** | A4 |
| **向き** | 縦（Portrait） |
| **呼び出し元画面** | BAT-002（バッチ実行履歴一覧画面）から「日次集計レポート出力」ボタンで呼び出し |

> 入力パラメータ（検索条件）は [API-10-report.md](API-10-report.md) の API-RPT-017 を参照。

## 2. PDFレイアウト

### 2.1. ヘッダー（1ページ目のみ追加表示する情報）

共通ヘッダー（[_standard-report.md](_standard-report.md) 参照）に加え、以下の情報を1ページ目に表示する。

| 要素 | 位置 | 内容 |
|------|------|------|
| 対象営業日 | 左 | `businessDate`（例: `2026-03-14`） |
| 対象倉庫数 | 右 | 「対象倉庫: {N}件」（レスポンス配列の要素数） |

> 抽出条件サマリーには「対象営業日: 2026-03-14」を表示する。倉庫の絞り込みは行わず全倉庫を出力する。

### 2.2. セクション構成

本レポートは通常の1テーブル形式ではなく、倉庫ごとに以下の5セクションを繰り返す構成とする。

```
倉庫ごとのブロック:
  ┌─ 倉庫ヘッダー（倉庫名）
  ├─ セクション1: 入荷サマリー
  ├─ セクション2: 出荷サマリー
  ├─ セクション3: 返品サマリー
  ├─ セクション4: 在庫サマリー
  └─ セクション5: 未処理アラート
```

### 2.3. 倉庫ヘッダー

各倉庫ブロックの先頭に倉庫名を太字・大きめ（11pt Bold）で表示する。

```
■ 東京DC（倉庫ID: 1）
```

### 2.4. セクション1: 入荷サマリー

セクションタイトル: 「入荷実績」（10pt Bold、下線付き）

| # | 項目名 | APIフィールド | 幅(mm) | 配置 | 書式 | 備考 |
|---|--------|-------------|--------|------|------|------|
| 1 | 入荷件数 | `inboundCount` | 95 / 95 | 項目名: 左 / 値: 右 | 整数 | 入庫完了伝票数 |
| 2 | 入荷明細行数 | `inboundLineCount` | 95 / 95 | 項目名: 左 / 値: 右 | 整数 | |
| 3 | 入荷数量合計 | `inboundQuantityTotal` | 95 / 95 | 項目名: 左 / 値: 右 | 整数 | バラ換算 |

> レイアウト形式: 2列のキー・バリュー形式（左に項目名、右に値）。幅合計: 95 + 95 = 190mm（A4縦の印字可能幅 190mm）

| # | 項目名 | 導出方法 | 書式 |

| 3 | 出荷件数合計 | 全倉庫の `outboundCount` を合算 | 整数 |
| 4 | 出荷数量合計 | 全倉庫の `outboundQuantityTotal` を合算 | 整数 |

| 6 | 返品数量合計 | 全倉庫の `returnQuantityTotal` を合算 | 整数 |
| 7 | 在庫総数量合計 | 全倉庫の `inventoryQuantityTotal` を合算 | 整数 |
| 8 | 未入荷件数合計 | 全倉庫の `unreceivedCount` を合算 | 整数 |
| 9 | 未出荷件数合計 | 全倉庫の `unshippedCount` を合算 | 整数 |

### 2.10. ページブレークルール

| ルール | 内容 |
|--------|------|
| 倉庫ブロック改ページ | 1倉庫ブロック（5セクション）が現在ページに収まらない場合、倉庫ブロックごと改ページする |
| 全倉庫合計改ページ | 全倉庫合計セクションが現在ページに収まらない場合、改ページして新ページに出力する |
| 倉庫ブロック分割防止 | 1つの倉庫ブロックはページをまたがない（必ず同一ページ内に収める） |

## 4. 特記事項

- **複数セクション構成**: 本レポートは通常の明細テーブル形式ではなく、倉庫ごとに5セクション（入荷・出荷・返品・在庫・未処理アラート）のキー・バリュー形式で構成される。テンプレートHTML側ではThymeleafの `th:each` で倉庫ブロックを繰り返し出力する
- **データソース**: 日替処理で生成された `inbound_summaries`、`outbound_summaries`、`return_slips`、`inventory_snapshots`、`unreceived_list_records`、`unshipped_list_records` の6テーブルから集計。データ取得仕様の詳細は [API-10-report.md](API-10-report.md) の API-RPT-017 業務ロジックを参照
- **日替処理完了前提**: 対象営業日の日替処理が `SUCCESS` で完了していない場合はAPIが 404 を返すため、レポート生成不可。画面側でエラーメッセージを表示する
- **未処理アラートの強調**: 未入荷件数または未出荷件数が1件以上の場合、該当行の値を赤色（`#D32F2F`）太字で表示する。0件の場合は通常色で表示する
- **全倉庫合計セクション**: 倉庫が1件のみの場合でも全倉庫合計セクションは表示する（将来の複数倉庫対応を見越して）
- **数量の単位**: `inboundQuantityTotal`、`outboundQuantityTotal`、`returnQuantityTotal`、`inventoryQuantityTotal` はすべてバラ（PCS）換算の合計値。各項目名の末尾に「（バラ換算）」を付記する
- **データ0件**: 対象営業日に全倉庫で入出荷実績がなく、在庫スナップショットのみ存在する場合でもレポートは生成される。入荷・出荷の件数は 0 と表示される
- **テンプレートファイル配置**: `backend/src/main/resources/templates/reports/RPT-17.html`
