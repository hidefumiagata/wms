# 非機能要件定義

> 本ドキュメントは WMS ShowCase プロジェクトの非機能要件を定義する。各要件の実現方式はアーキテクチャブループリントの各セクションで詳細化されている。

---

## 1. 性能要件

| 項目 | 要件値 | 備考 |
|------|--------|------|
| **API応答時間（通常操作）** | 2秒以内 | 3秒超過でアラート発報（[11-monitoring-operations.md](11-monitoring-operations.md)） |
| **API応答時間（集計・レポート系）** | 5秒以内 | 日次集計・PDF出力等 |
| **画面初期表示** | 3秒以内 | Blob 静的ホスティングからの配信 |
| **バッチ処理（日替処理）** | 30分以内 | 5処理の合計実行時間 |
| **同時接続ユーザー数** | dev: 10名、prd: 50名 | Container Apps の自動スケールで対応 |

---

## 2. 可用性要件

> prd 環境は Terraform Deploy/Destroy モデルで運用する。稼働率は**稼働中の場合**の目標値。

| 項目 | 要件値 | 備考 |
|------|--------|------|
| **目標稼働率（prd 稼働中）** | 99.5%以上 | 計画停止（Destroy/Deploy）を除く |
| **計画停止** | 月次メンテナンス最大2時間 | 業務時間外に実施 |
| **業務稼働時間帯** | 平日 7:00〜22:00 | 倉庫業務対応時間帯 |
| **RTO（フロント/API）** | 数分 | Azure Front Door による自動フェイルオーバー（prd のみ） |
| **RTO（DB復旧）** | 数時間 | 手動対応（ポイントインタイムリストア） |
| **RPO** | 最大1時間 | Geo-redundant backup 基点（prd のみ） |

> 詳細は [06-infrastructure-architecture.md](06-infrastructure-architecture.md) の prd 環境フェイルオーバー仕様を参照。

---

## 3. スケーラビリティ要件

| 項目 | 内容 |
|------|------|
| **水平スケール（バックエンド）** | Container Apps の HTTP リクエスト数トリガーによる自動スケール |
| **スケール設定（dev）** | min: 0、max: 3（未使用時ゼロコスト） |
| **スケール設定（prd East）** | min: 1、max: 5（常時1台待機） |
| **スケール設定（prd West）** | min: 0、max: 5（スタンバイ） |
| **データ増加対応** | 完了済み2か月超のトランデータを日替処理でバックアップテーブルに移行し、主テーブルの肥大化を抑制 |

---

## 4. セキュリティ要件

> 詳細は [07-auth-architecture.md](07-auth-architecture.md) および [10-security-architecture.md](10-security-architecture.md) を参照。

| 項目 | 要件 |
|------|------|
| **認証方式** | JWT（httpOnly Cookie）+ Spring Security |
| **認可方式** | ロールベースアクセス制御（RBAC）。SYSTEM_ADMIN / WAREHOUSE_MANAGER / WAREHOUSE_STAFF / VIEWER の4ロール |
| **セッションタイムアウト** | 操作なし1時間で自動ログアウト（リフレッシュトークンのスライディング方式） |
| **パスワードポリシー** | 8文字以上。BCrypt ハッシュ保存 |
| **初回ログイン** | パスワード変更を強制（変更完了まで他操作不可） |
| **ログイン失敗ロック** | 連続5回の失敗でアカウントロック。解除は SYSTEM_ADMIN のみ |
| **通信暗号化** | HTTPS 必須（TLS 1.2以上）。HTTP は HTTPS へリダイレクト |
| **WAF** | Azure Front Door WAF Standard（prd 環境のみ） |
| **入力バリデーション** | フロントエンド（Element Plus）＋バックエンド（Jakarta Bean Validation）の二重チェック |
| **SQLインジェクション対策** | Spring Data JPA パラメータバインディング。ネイティブクエリは `:param` バインド必須 |
| **セキュリティヘッダー** | `X-Content-Type-Options: nosniff`、`X-Frame-Options: DENY`、`Referrer-Policy: strict-origin-when-cross-origin` |
| **ネットワーク分離** | Container Apps・PostgreSQL をプライベートサブネット（VNet 統合）に配置 |

---

## 5. 信頼性要件

| 項目 | 内容 |
|------|------|
| **データ整合性** | DB トランザクション管理（Spring @Transactional）、楽観的ロック（更新競合検出） |
| **論理削除** | マスタデータは物理削除せず有効/無効フラグで管理 |
| **バッチ二重実行防止** | 同一営業日の処理済みフラグを DB で管理し、二重実行をエラーで防止 |
| **DB 自動バックアップ** | Azure PostgreSQL Flexible Server の自動バックアップ（7日間保持） |
| **業務データアーカイブ** | 完了状態かつ2か月以上前のトランデータを DB バックアップテーブルに無期限保持 |
| **マルチリージョン冗長（prd）** | Japan East（Active）/ Japan West（Standby）の Active-Passive 構成 |

---

## 6. 保守性・運用性要件

| 項目 | 内容 |
|------|------|
| **アプリケーションログ** | JSON 構造化ログ（INFO / WARN / ERROR）。traceId 付きでリクエスト追跡可能 |
| **操作ログ（監査証跡）** | 在庫訂正・ユーザー操作の実施者・日時を記録 |
| **認証ログ** | ログイン成功・失敗・アカウントロックを INFO/WARN レベルで出力 |
| **ログ保存期間** | Log Analytics Workspace: 30日（無料枠内） |
| **監視・アラート** | Azure Monitor + Log Analytics。ERROR ログ検知・API 遅延・DB 接続失敗・ログイン失敗多発をメール通知 |
| **デプロイ方式** | GitHub Actions CI/CD → Azure Container Registry → Container Apps |
| **IaC** | Terraform（Deploy/Destroy モデル）。dev / prd 環境を独立管理 |
| **DB スキーマ管理** | Flyway マイグレーション。バージョン管理下でスキーマ変更を適用 |

---

## 7. ユーザビリティ要件

| 項目 | 内容 |
|------|------|
| **対応デバイス** | デスクトップブラウザのみ（モバイル・タブレット対応はスコープ外） |
| **対応ブラウザ** | Chrome 最新版、Microsoft Edge 最新版 |
| **対応解像度** | 1280×768 以上 |
| **多言語対応** | 日本語・英語（vue-i18n）。UI 言語を切り替え可能 |
| **UIコンポーネント** | Element Plus 準拠（アクセシビリティ・操作性は Element Plus 標準に従う） |
| **倉庫切り替え** | ヘッダーのプルダウンで選択中倉庫を常設表示。全業務画面がフィルタされる |

---

## 8. 法令・コンプライアンス要件

| 項目 | 内容 |
|------|------|
| **個人情報管理** | 取引先担当者情報（氏名・電話番号・メールアドレス）をシステム内に保持。アクセスは認証済みユーザーのみ |
| **業務データ保持** | トランザクションデータは無期限保持（アーカイブ含む） |
| **監査証跡** | 在庫訂正・ユーザー操作の履歴を記録。誰が・いつ・何を操作したかを追跡可能 |

---

## 9. 移行・拡張性要件

| 項目 | 内容 |
|------|------|
| **外部 I/F** | CSV インポート（入荷予定取込）。Azure Blob Storage の `iffiles` コンテナ経由 |
| **将来の API 連携** | RESTful API（JSON）設計。外部システムとの直接 API 連携を将来的に追加できる構造 |
| **複数倉庫対応** | 倉庫切り替え機能により、複数倉庫を1システムで管理 |
| **初期データ投入** | Flyway SQL スクリプトで投入（CSV インポート機能は不要） |
