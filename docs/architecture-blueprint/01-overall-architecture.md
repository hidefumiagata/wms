# 全体アーキテクチャ方針

## アーキテクチャスタイル

**モジュラーモノリス** を採用する。

| 選択肢 | 採否 | 理由 |
|--------|------|------|
| モノリス | ✗ | 将来の拡張性が低い |
| マイクロサービス | ✗ | 1人開発にはオーバーヘッドが大きく、コスト制約にも合わない |
| **モジュラーモノリス** | ✅ | 開発の簡潔さと将来のマイクロサービス移行可能性を両立 |

## モジュール構成

| モジュール | 英名 | 主な責務 |
|-----------|------|---------|
| 入荷 | `inbound` | 入荷予定・入荷検品・入庫指示 |
| 在庫 | `inventory` | ロケーション管理・棚卸・在庫照会 |
| 在庫引当 | `allocation` | 受注データを元に出荷分在庫を引き当て・ピッキング可能化 |
| 出荷 | `outbound` | ピッキング指示・出荷検品・配送指示 |
| マスタ | `master` | 商品・取引先・ロケーション・ユーザ |
| レポート | `report` | 各種帳票・一覧出力 |
| バッチ | `batch` | 日替処理・各種集計 |
| 外部連携 | `interface` | 入荷予定/受注取り込みI/F |
| 共通基盤 | `shared` | ロギング・エラーハンドリング・バリデーション等 |
