# 全体アーキテクチャ方針

## アーキテクチャスタイル

**モジュラーモノリス** を採用する。

| 選択肢 | 採否 | 理由 |
|--------|------|------|
| モノリス | ✗ | 将来の拡張性が低い |
| マイクロサービス | ✗ | 1人開発にはオーバーヘッドが大きく、コスト制約にも合わない |
| **モジュラーモノリス** | ✅ | 開発の簡潔さと将来のマイクロサービス移行可能性を両立 |

## モジュール構成

| モジュール | 英名 | 主な責務 |
|-----------|------|---------|
| 入荷 | `inbound` | 入荷予定・入荷検品・入庫指示 |
| 在庫 | `inventory` | ロケーション管理・棚卸・在庫照会 |
| 在庫引当 | `allocation` | 受注データを元に出荷分在庫を引き当て・ピッキング可能化 |
| 出荷 | `outbound` | ピッキング指示・出荷検品・配送指示 |
| マスタ | `master` | 商品・取引先・ロケーション・ユーザ |
| レポート | `report` | 各種帳票・一覧出力 |
| バッチ | `batch` | 日替処理・各種集計 |
| 外部連携 | `interface` | 入荷予定/受注取り込みI/F |
| システム共通 | `system` | 営業日取得API等のシステム横断共通機能 |
| 共通基盤 | `shared` | ロギング・エラーハンドリング・バリデーション等 |

## システム概念：営業日

WMSでは、カレンダー上のシステム日付（今日）とは別に「**営業日**」という業務上の日付概念を持つ。

| 項目 | 内容 |
|------|------|
| **定義** | 日替処理（バッチ）が実行されるまで変わらない業務上の日付 |
| **更新タイミング** | `POST /api/v1/batch/daily-close`（日替処理）の実行時に翌日付へ進む |
| **業務上の意味** | 「今日」「当日」「本日」はすべて「現在営業日」を指す |
| **UI表示** | 全画面のヘッダーに現在営業日を常時表示する |
| **管理場所** | `system_parameters` テーブル（または専用の `business_date` テーブル）で一元管理 |

> **例：** 営業日が 2026-03-13 の状態で日替処理を実行すると、営業日が 2026-03-14 に進む。
> 日替処理を実行しない限り、たとえカレンダー上で日付が変わっても営業日は変わらない。
